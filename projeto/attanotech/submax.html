Submax

Stack e arquitetura

  • Backend: Django + Django REST Framework.
  • Banco de dados: MySQL.
  • #Frontend Frontend: HTML templates Django (ou SPA leve com JS) com Bootstrap 5.
  • Requisições assíncronas: Fetch API (ou Axios).
  • Dados de rotina: salvar como JSON no banco (campo JSON ou Text + validação).
  • APIs internas:
    • /api/alunos/ (listagem, busca, filtros)
    • /api/alunos/<id>/ (detalhe)
    • /api/exercicios/?q= (busca paginada por nome)
    • /api/rotinas/ (CRUD)
    • /api/rotinas/<id>/ (CRUD)
  • Padrão de logs para debug: logs estruturados no Django (logger por app) e endpoints com tratamento de erro consistente.

Módulo 1 — Alunos (tela “Alunos”)

Crie uma tela /alunos com:

  • Botão “+ Novo Aluno”
  • Botão “Importar Excel”
  • Campo de busca “Buscar Aluno…”
  • Dropdown de status (Ativos/Inativos/…)
  • Tabela com colunas: Nome | Plano | Status do Plano | Telefone | Email | Ações
  • Ação principal: clicar na linha do aluno abre /alunos/<id>.

Módulo 2 — Detalhe do Aluno (tela “Detalhe - Aluno”)

Página /alunos/<id> contendo:

  1. Card de perfil do aluno (foto, nome, email, telefone).
  2. Cards de Preferências e Informações Médicas (campos simples).
  3. Gráficos:
    • “Grupos Musculares por Mês” (barra/coluna) com filtro de mês.
    • “Avaliações no Tempo” (linha) com séries (Peso, Massa Muscular, Gordura).
  4. Seção Rotina de Treinos com botão “+ Nova Rotina”.
  5. Seção Avaliações com botão “+ Nova Avaliação”.

Módulo 3 — Rotina de Treinos (Editor em modal/página)

Ao criar/editar rotina, seguir este fluxo:

3.1 Criação da rotina

Ao clicar “+ Nova Rotina”:

  • Mostrar modal/passo 1: solicitar (a) quantidade de semanas OU (b) data início e data fim.
  • Se o usuário informar datas: calcular automaticamente a quantidade de semanas (arredondar para cima; semana = 7 dias).
  • Gerar o modelo inicial da rotina com:
    • nome da rotina (default: “Hipertrofia - Bloco A” editável)
    • data_inicial, data_final

3.2 UI por abas (nav-tabs)

No editor:

  • Criar nav-tabs de semanas (Semana 1, Semana 2, …).
  • Dentro de cada semana, permitir múltiplas nav-tabs de séries:
    • Botão “+ Adicionar Série”
    • Cada série:
      • nome editável inline (ex.: “Série A”)
      • reordenável (drag and drop ou botões “↑ ↓”)
  • Ao selecionar Semana + Série, exibir:
    • Barra de busca para exercícios (autocomplete via API /api/exercicios/?q=).
    • Botões: “Submáxima”, “Edição Avançada”, “+ Adicionar Exercício”.

3.3 Tabela de exercícios dentro da série

A tabela deve ter colunas:

  • Checkbox
  • Exercício
  • Série (número)
  • Repetições
  • Carga
  • Intervalo
  • Método

Regras:

  • Checkbox habilita seleção múltipla.
  • “Edição Avançada” abre um painel/modal para aplicar valores em lote aos exercícios selecionados:
    • séries, repetições, intervalo, método, e também ajuste de carga se aplicável.
  • Coluna “Método”:
    • exibir select com métodos de treino (ex.: Tradicional, Drop-set, Bi-set, Rest-pause, Pirâmide, etc.)
    • ao selecionar um método, abrir uma linha abaixo do exercício (row expand/collapse) com descrição dinâmica de “como executar”.
    • a descrição deve ser dependente do método escolhido.

3.4 Submáxima (cálculo de carga)

Criar um botão “Submáxima” que calcula carga sugerida com:

  • Fórmula linear:
    • LINEAR_A = 0.0375
    • LINEAR_B = 0.978
  • Existe uma tabela SUBMAXIMA_PERCENTUAIS que mapeia repeticoes -> percentual
    Exemplo: 1→100%, 2→95%, …, 15→65%
  • PERCENTUAIS_KEYS guarda as chaves da tabela em ordem.
    Implementar comportamento:
  • Usuário informa uma carga de referência (ex.: 1RM ou carga base) e a repetição alvo.
  • O sistema busca percentual na tabela por repetição (se não existir, usar aproximação pela chave mais próxima em PERCENTUAIS_KEYS).
  • Aplicar fórmula linear como ajuste final (documentar no código).
  • Preencher a coluna Carga automaticamente (ex.: “60kg”).
  • Permitir aplicar submáxima:
    • somente no exercício atual
    • ou em lote via Edição Avançada (para itens selecionados)

3.5 Bodygraph + Gráfico de volume por músculo

Ao adicionar exercícios via API, cada exercício retorna:

  • grupo muscular primário
  • grupos musculares secundários
    Esses dados alimentam:
  • Um bodygraph (imagem/ícone do corpo com destaque nos músculos) baseado nos grupos usados.
  • Um gráfico de barras “Volume por grupo muscular” (contagem/peso por músculo).
    Regras:
  • Atualizar em tempo real ao adicionar/remover exercício ou mudar série/repetições.
  • O cálculo do “volume” pode ser:
    • simples (ex.: séries * repetições)
    • e opcional ponderado pela carga quando existir (ex.: séries repetições carga_num)
      Escolher um padrão e deixar a função pronta para evoluir.

Modelagem de dados (Django)

Criar models mínimos:
Aluno

  • nome, email, telefone, foto (opcional), status, plano, etc.

Exercicio

  • nome
  • grupo_muscular_primario (FK ou CharField)
  • grupos_musculares_secundarios (ManyToMany ou JSON simples)
  • (opcional) metadata

RotinaTreino

  • aluno (FK)
  • nome
  • data_inicial
  • data_final
  • estrutura (JSONField)
  • created_at / updated_at

Estrutura JSON salva (obrigatório)

Salvar exatamente neste formato base (permitindo campos extras):

estrutura = {
  "semanas": [
    {
      "nome": "Semana 1",
      "series": [
        {
          "nome": "Série A",
          "exercicios": [
            {
              "nome": "Agachamento Livre",
              "series": 4,
              "repeticoes": 10,
              "carga": "60kg",
              "intervalo": "90s",
              "metodo": "Tradicional",
              "metodo_descricao": "..."
            }
          ]
        }
      ]
    }
  ]
}

UX e comportamento

  • Tudo deve funcionar com salvamento incremental:
    • botão “Salvar” salva a estrutura inteira.
    • mas também salvar automaticamente (debounce) em mudanças importantes (opcional).
  • Validações:
    • data_final >= data_inicial
    • semanas >= 1
    • séries e repetições numéricas > 0
  • Performance:
    • busca de exercícios paginada e com debounce.
  • Acessibilidade:
    • atalhos e foco no campo de busca ao selecionar série.
  • Mensagens:
    • toast de sucesso/erro.
    • tratamento para falha na API (mostra erro e mantém estado do formulário).
