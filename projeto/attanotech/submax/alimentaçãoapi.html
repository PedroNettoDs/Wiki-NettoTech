1) /api/alunos/ (AlunoViewSet)

Auth

✅ IsAuthenticated

Operações

  • GET /api/alunos/
    Filtros (querystring):
    • ativo, sexo, plano_status, plano_tipo
    • search= (busca em nome, email, telefone)
    • ordering= (ex.: nome, -criado_em, data_nascimento, plano_tipo__nome)
  • POST /api/alunos/
    Body (JSON): o serializer define os “required”, mas pelo uso na view dá pra afirmar que suporta:
    • nome (provavelmente obrigatório)
    • data_nascimento, telefone, email, sexo
      • vários campos de anamnese/preferências (os mesmos retornados em detalhes)
    Efeito interno importante: no perform_create, ele salva id_usuario=request.user (ou seja, “dono” do aluno).
  • GET /api/alunos/{id}/ (retrieve)
  • PUT/PATCH /api/alunos/{id}/ (update)
  • DELETE /api/alunos/{id}/ (destroy)

Endpoint extra

  • GET /api/alunos/{id}/detalhes/
    Retorna um JSON consolidado + calcula idade usando data_nascimento.
  • POST /api/alunos/{id}/foto/
    multipart/form-data com arquivo no campo foto (aceita fallback file ou imagem).
    Processa com PIL, converte pra JPG e salva base64 em aluno.foto_b64.
    Retorna foto_b64 + foto_url (se o serializer expõe).
  • POST /api/alunos/{id}/remover_foto/
    Zera foto_b64.

3) /api/avaliacoes/ (AvaliacaoViewSet)

Auth

✅ IsAuthenticated
✅ Escopo por dono via owner_lookup = aluno__id_usuario (e também permite “legado” quando aluno não tem dono).

Operações

  • GET /api/avaliacoes/
    Query params:
    • aluno_id= ou aluno= (filtra)
    • filtros: aluno, data
    • search= em aluno__nome, observacoes
    • ordering= em data, peso, altura
    • paginação page=
  • POST /api/avaliacoes/
    multipart/form-data:
    • Campos de avaliação (JSON-like) via form (ex.: aluno, data, peso, altura, observacoes etc — depende do serializer)
    • Upload múltiplo de fotos em fotos (pode enviar várias)
    Efeito interno: ele remove fotos do request.data para o serializer (para não validar como campo comum) e trata os arquivos via _save_images().
    As fotos ficam em Avaliacao.fotos (JSONField) com chaves tipo ID1, ID2… guardando base64.
  • PUT/PATCH /api/avaliacoes/{id}/
    Também aceita anexar fotos via fotos (multipart) e ainda suporta remoção:
    • remove_fotos=ID1 ou remove_fotos=1
    • pode vir múltiplo (getlist) ou string CSV/JSON.

Endpoint extra (servir imagem)

  • GET /api/avaliacoes/{id}/foto/{key}/
    key pode ser ID1 ou 1 (compat com lista legada).
    Retorna image/jpeg.

6) /api/agendas/ (AgendaViewSet) + feeds de calendário

Auth

⚠️ No código desse ViewSet não tem permission_classes.
Como o REST_FRAMEWORK final do settings.py também não define permissões default, esse endpoint pode estar público (a não ser que exista middleware/restrição fora do DRF). Vale conferir.

Operações

CRUD padrão (campos dependem do serializer), mas pelo feed dá pra inferir que Agenda tem pelo menos:

  • titulo, data, hora_inicio, hora_fim, descricao, tipo, aluno, personal, link_externo.

Feed unificado (FullCalendar)

  • GET /agendas/api/calendar/feed?start=YYYY-MM-DD&end=YYYY-MM-DD&aluno=<id>&personal=<id>
    • Filtra agendas por data range, aluno, e personal (direto ou via relacionamento aluno__personais).
    • Também adiciona “lembretes” de próxima avaliação de Aluno.avaliacao_proxima_em (allDay).
